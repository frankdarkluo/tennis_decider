## 本周最重要的进展
- 诊断链路完成从解析、排序到默认层文案预算的全面收口：混合输入稳定保留技术主问题，quick / standard / deep 主线一致，低证据场景默认输出更短且聚焦可执行动作。
- Deep diagnosis 架构从"平行 prototype 页面"收敛为 `/diagnose` 内的 inline subflow，`EnrichedDiagnosisContext` 贯穿 diagnosis → plan handoff，serve gold slice 的 7 天计划已全程保持场景特异性。
- 研究执行闭环基础夯实：flush 失败分桶、诊断 snapshot 可回放、真实样本回归集、文案预算机制、结果一致性约束全部落地；docs 结构也完成了一次从目录到链接的系统性重组。

## 功能推进

### 诊断质量（Mar 30–Apr 1）
- 混合输入判别、子句覆盖奖励、技术/心理冲突惩罚迭代，减少误判到兜底心理标签。
- 加入 `summary` 预算与 `detailedSummary` 分层，降低默认层信息密度；展开行为保留 `why_this_viewed` 事件打点。
- 推荐列表增加创作者去集中化与场景直链优先，计划评分更依赖诊断上下文词命中。
- 内容库从 673 扩到 1065，legacy canonical 缺失与高集中簇告警清零。

### Diagnosis → Plan 主线收紧（Apr 2）
- `primaryNextStep` 现在稳定贯穿 plan summary、target、Day 1 focus / goal / success criteria，用户从诊断进入计划时始终看到同一条下一步。
- 补充 3 条高价值口语诊断规则：`发球老是扔不准球`、`正手怎么抡都不走球`、`接发一上来就被顶回去`。
- 修复多处 React hydration mismatch；移除研究流程中不需要的 Actionability 提示渲染（保留预览页与测试覆盖）。

### Deep Diagnosis + Scenario Reconstruction（Apr 4）
- 场景还原原型完成最小闭环（自由文本 → 结构化提取 → follow-up 追问 → 完成态 → handoff），随后收口到 `/diagnose` 作为 deep flow 唯一 orchestrator。
- 引入 `EnrichedDiagnosisContext`，携带 stroke family、serve subtype、session type、pressure context 等字段，贯穿 deep diagnosis → plan handoff。
- `serve` gold slice 的 deep plan 重写全 7 天，按明确 progression ladder 推进（Day 1 baseline → Day 7 carry-forward rule），消除 late-week generic template decay。
- 补充 deterministic golden fixtures（PR5），锁定 Day 4–7 的场景特异性，防止版本漂移。

### Docs 结构重组（Apr 5）
- `docs/` 按用途分组：`product/`、`engineering/`、`roadmap/archive/` 新增；research 文件改为稳定名称；所有内部 wikilink 同步更新。
- 新增 `docs/README.md` 说明结构规则与归档约定。

## 关键决策
- ==诊断默认层优先保"可执行的主动作"==，详细解释按需展开，而不是默认堆叠长文案。
- quick / standard / deep 允许解释深度不同，但不允许 `problemTag` 和 `primaryNextStep` 漂移。
- Deep mode 收敛为 `/diagnose` 内的增强流程，不拆第二套结果页或计划页。
- 观测增强优先做在聚合与导出层，不更改既有 study event schema。

## 本周形成的结论
- 本周的真实主线是把研究模式从"可用"收口到"更稳、更短、更可验证"：诊断方向更稳，计划更可执行，研究日志更可检查，深入模式架构更清晰。
- Deep diagnosis 最重要的架构判断是"单一 orchestrator 优于平行页面"：不拆第二套 UI，不引入 runtime LLM，而是用 deterministic overlay 让现有框架承载更深的上下文。
- 内容扩库只有和清洗、直链过滤、候选策略、文案压缩一起完成，才真正提升 routing 质量；单纯堆数量不是核心价值。
