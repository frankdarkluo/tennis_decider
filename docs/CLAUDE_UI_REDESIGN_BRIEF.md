# TennisLevel UI Simplification Brief

## 这份文档的目的

这是一份给 Claude Opus 的产品与设计说明，目标不是复述全部代码，而是帮助它快速理解：

1. 当前各页面的 UI 结构和信息组织方式
2. 为什么现在会显得“重”
3. 如果要往更轻、更短、更像 Keep 的方向走，优先该改什么
4. 哪些代码文件最值得先动

整体方向可以概括成一句话：

> 把 TennisLevel 从“信息很多的工具页”收成“每次只带用户做一件事的轻量教练页”。

---

## 整体设计判断

### 现在产品的优点

- 功能已经比较全：水平评估、文字诊断、视频诊断、内容库、博主榜、训练计划都打通了
- 推荐逻辑已经有一定“教练视角”，不是纯平台搜索
- 内容、博主、训练计划之间已经能互相跳转

### 现在产品显得重的主要原因

- 很多页面同时出现太多模块，用户一进来就要读很多字
- 同一页里经常既有说明、又有列表、又有按钮、又有补充入口，层级太平
- “主任务”不够突出，用户容易分心
- 有些页面把“全部信息一次性展开”了，没有分层展示
- 文字偏多，但视觉辅助偏少，所以阅读疲劳感比较明显

### 建议的总原则

参考 Keep 的感觉，不是照搬风格，而是借它的几个核心方法：

1. 一屏只做一件事  
   每个页面都应该有一个最主要动作，其他动作全部降级。

2. 一次只给用户一个判断  
   不要一口气告诉用户 5 个问题、6 个原因、7 个建议。先给“最关键的一个”。

3. 用短句代替段落  
   标题尽量口语化，说明尽量控制在 1 到 2 行。

4. 默认收起次要信息  
   先给结论，展开后再看原因、细节、备选内容。

5. 用视觉帮助理解，而不是只靠文字  
   可以用简单的小人动图、动作小图标、步骤卡片、进度条来替代一部分解释文字。

## 各页面 UI 现状与建议

## 1. 首页

### 当前页面在做什么

首页现在已经比较完整，主要由这几块组成：

- Hero：一句话提问 + 常见问题标签 + 立即诊断
- 快速入口
- 提升路径说明
- 热门内容
- 热门博主
- 底部再引导去水平评估

关键代码：

- `src/app/page.tsx`
- `src/components/home/HeroSection.tsx`

### 当前为什么会显得有点重

- 首页既像 landing page，又像导航页，又像内容聚合页
- Hero 区域下面有多个次级入口，主次关系不够鲜明
- 用户刚打开页面时，其实最需要的是“我现在该从哪里开始”

### 更轻的改法

建议首页只保留 3 个重点区块：

1. 顶部主入口  
   一个大输入框或一句话入口，只强调“说出你的问题”。

2. 两个次级入口  
   一个是“先做水平评估”，一个是“上传视频诊断”。

3. 精选推荐  
   首页底部只保留少量高价值内容，例如 3 位博主 + 3 条精选内容，不要像完整内容页一样展开。

### 文案风格建议

- 现在：`一句话，帮你找到下一步该练什么`
- 可以更口语：`告诉我你最近最卡的一球`

- 现在：`直接描述你最近在网球里遇到的问题，我们会给你更像教练的诊断、内容推荐和训练方向。`
- 可以更短：`你说问题，我来帮你判断先练什么。`

### 更值得优先改的代码点

- `src/components/home/HeroSection.tsx`
  - 减少次级链接数量
  - 常见问题标签从 5 个减到 3 个
  - 让主按钮更突出

- `src/app/page.tsx`
  - 压缩首页模块数量
  - 只保留最强的 2 到 3 个下游区块

## 2. 水平评估

### 当前页面在做什么

水平评估现在是一个多题 step-by-step 流程，用户逐题选择，最后得到等级结果。

关键代码：

- `src/app/assessment/page.tsx`
- `src/components/assessment/QuestionCard.tsx`
- `src/components/assessment/AssessmentProgress.tsx`

### 当前为什么会显得重

- 题目数量偏多，用户会有“还没做完”的压力
- 每一题的选项都要认真读，整体阅读量不小
- 当前的体验更像传统问卷，而不是轻量自测

### 更轻的改法

建议把评估从“完整问卷”改成“快速判断”：

1. 题目从 8 题压到 4 到 5 题  
   只保留最能区分水平的题，比如：
   - 发球能不能稳定发进
   - 正反手能不能连续对拉
   - 脚步和站位是否有意识
   - 比赛里能不能稳定组织一分

2. 每题选项改成更口语、更短  
   不要像考试题，要像教练提问。

3. 减少“读文字”的负担  
   可以把每题做成“1 句提问 + 3 个很短的选项卡片”。

4. 更早给反馈  
   做到第 3 题时，就可以给一个“你大概在 3.0 附近”的轻提示。

### 适合加入的视觉辅助

- 每个题目配一个小动作图标
- 进度条保留，但弱化“第几题”，强化“快完成了”
- 结果页可以用一个简化的小人练球动画来表达“当前水平的典型状态”

### 更值得优先改的代码点

- `src/components/assessment/QuestionCard.tsx`
  - 把一整列纯文字按钮改成更轻的卡片式选择

- `src/components/assessment/AssessmentProgress.tsx`
  - 从“第 X 题 / 共 Y 题”改成更轻的进度表达

- `src/app/assessment/page.tsx`
  - 直接减少题目总数
  - 允许中途给出预估结果

## 3. 问题诊断

### 当前页面在做什么

文字诊断页允许用户描述问题，然后返回：

- 诊断标题
- 一段 summary
- 分类和置信度
- 命中的关键词
- 可能原因
- 优先修正点
- 推荐内容
- 去训练计划
- 去视频诊断
- 平台搜索补充

关键代码：

- `src/app/diagnose/page.tsx`
- `src/components/diagnose/DiagnoseResult.tsx`

### 当前为什么会显得重

- 一个结果卡里承载了太多层级的信息
- “原因”“修正点”“推荐内容”“下一步动作”都同时出现
- 用户刚做完诊断时，最需要的是一个清楚的“先改什么”，而不是一次读很多段

### 更轻的改法

建议把结果结构改成三层：

1. 第一层：一句话结论  
   例如：`你现在最该先改的是：反手击球点太晚`

2. 第二层：一个立即动作  
   例如：`今天先记住这一个修正：提前准备，别等球到身边再挥拍`

3. 第三层：展开看更多  
   点开后再显示：
   - 为什么会这样
   - 推荐内容
   - 平台补充搜索

### 更建议的页面节奏

- 输入页：更像聊天入口
- 结果页：更像教练一句话点评
- 次级内容：放进折叠区或次级卡片

### 更值得优先改的代码点

- `src/components/diagnose/DiagnoseResult.tsx`
  - `causes` 不要默认全部展开
  - `fixes` 默认只显示第 1 条
  - `recommendedContents` 默认只显示 1 到 2 条
  - `PlatformVideoSearch` 放到折叠区或页面下半段

- `src/app/diagnose/page.tsx`
  - 输入完成后，别再同屏出现太多“继续追问”式内容

## 4. 视频诊断

### 当前页面在做什么

视频诊断页现在功能很强，但也最重。它同时包含：

- 击球类型选择
- 场景选择
- 补充描述
- 视频上传
- 上传状态 / 处理中状态
- 使用次数提示
- AI 观察结果
- 问题原因
- 推荐内容
- 推荐博主
- 训练计划
- 搜索建议

关键代码：

- `src/app/video-diagnose/page.tsx`
- `src/components/video/VideoAnalysisResult.tsx`

### 当前为什么会显得重

- 这是一个典型“单页承载太多信息”的页面
- 用户需要先填、再传、再等、再读很多卡片
- 结果页里推荐内容、推荐博主、训练计划、搜索建议都一起给了

### 更轻的改法

建议把视频诊断改成明确的 3 步：

1. 选动作  
   只保留最必要的 2 到 3 个字段。

2. 上传视频  
   上传区做大，说明做短。

3. 看结果  
   默认只给：
   - 最核心问题
   - 一条修正建议
   - 一条最值得看的内容

剩下的“推荐博主”“搜索建议”“完整训练计划”全部降级。

### 结果页建议的层级

- 顶部：`先改这一个`
- 中部：`AI 为什么这样判断`
- 底部：`如果你还想继续看`

### 更值得优先改的代码点

- `src/components/video/VideoAnalysisResult.tsx`
  - 首屏只保留一个主问题 + 一个动作建议
  - `recommendedContents` 首屏只展示 1 条
  - `recommendedCreators` 改成折叠区域
  - `searchSuggestions` 可以不默认展示
  - 7 天训练计划只保留一句摘要 + 进入计划页按钮

- `src/app/video-diagnose/page.tsx`
  - 表单字段减少
  - 页面从“信息堆叠”改成“步骤式流程”

## 5. 内容库

### 当前页面在做什么

内容库已经是一个比较完整的筛选页，支持平台、语言、类型、方向、创作者、收藏等维度。

## 对你提出的 3 个需求的具体想法

## 1. 用户界面再精简，参考 Keep

### 我对这个方向的理解

你不是要“做成 Keep 的颜色或样式”，而是要它的这几个体验特征：

- 页面轻
- 字少
- 主动作清楚
- 不压迫用户
- 让人愿意马上点下一步

### 我建议直接执行的设计动作

1. 每页只保留一个主按钮  
   次按钮能变成文字链接的，就不要都做成大按钮。

2. 每页首屏只保留一个重点结论  
   比如诊断页只先说“先改这一个”，不要同屏给 4 块卡片。

3. 减少边框和卡片数量  
   不是每一段都需要一张卡。

4. 增加留白  
   比起继续堆内容，更有效的是让内容更有呼吸感。

5. 把说明写成一句话  
   不要一上来给用户解释整个系统。

### 最值得先动的文件

- `src/app/page.tsx`
- `src/components/home/HeroSection.tsx`
- `src/app/diagnose/page.tsx`
- `src/components/diagnose/DiagnoseResult.tsx`
- `src/app/video-diagnose/page.tsx`
- `src/components/video/VideoAnalysisResult.tsx`

---

## 2. 信息量过大，减少问答个数和阅读的信息量

### 我对这个方向的理解

现在最大的问题，不是“缺内容”，而是“用户第一次用时要读太多”。  
所以最有效的不是继续加功能，而是减少第一次必须处理的信息量。

### 我建议的具体做法

1. 水平评估从 8 题降到 4 到 5 题  
   只保留最有区分度的问题。

2. 问题诊断默认只给 1 个主要原因 + 1 个修正动作  
   其他原因折叠。

3. 视频诊断首屏只给 1 个问题 + 1 条内容 + 1 个下一步  
   不要一次给完博主、计划、搜索建议。

4. 内容库和博主榜默认只显示最常用筛选  
   其他项收进“更多筛选”。

5. 训练计划首屏只强调今天  
   Day 2 到 Day 7 降级。

### 一个很适合实现的代码思路

可以引入“分层展示”的通用模式：

```tsx
const [expanded, setExpanded] = useState(false);

return (
  <>
    <PrimarySummary />
    {expanded ? <SecondaryDetails /> : null}
    <button onClick={() => setExpanded((v) => !v)}>
      {expanded ? "收起细节" : "展开更多"}
    </button>
  </>
);
```

这个模式特别适合：

- `DiagnoseResult`
- `VideoAnalysisResult`
- `ContentCard`
- `DayPlanCard`

## 3. 文字 visualize，可以有小人动图

### 我对这个方向的理解

你想解决的是：  
有些动作类信息，只靠文字不够直观。  
如果能用轻量视觉去表达“准备太晚”“击球点太靠后”“分腿垫步”“发球 trophy pose”这些概念，用户理解速度会更快。

### 我建议不要一上来做太重的动画

不建议一开始就做复杂 3D 或大视频。  
更适合 TennisLevel 的是这几种轻量形式：

1. 循环 SVG 小人动图  
   轻、快、可控、适合网页。

2. Lottie 动画  
   适合做简短动作示意，比如“准备拍”“分腿垫步”“挥拍路径”。

3. 简单动作帧图  
   也可以不用真正动画，而是 3 帧小图加轻微切换。

### 我建议优先放的 4 个位置

1. 水平评估  
   每个问题旁边放一个小动作提示图。

2. 问题诊断结果  
   在“先改这一个”旁边放一个对应动作示意。

3. 视频诊断结果  
   把 AI 观察到的问题，用 1 个小动作动画表达。

4. 训练计划  
   每天的动作重点配一个小人图标或小动画。

### 很适合抽成一个通用组件

```tsx
type MotionCoachCueProps = {
  motion: "split_step" | "late_contact" | "trophy_pose" | "high_finish";
  label?: string;
};

export function MotionCoachCue({ motion, label }: MotionCoachCueProps) {
  return (
    <div className="flex items-center gap-3 rounded-2xl bg-slate-50 px-3 py-3">
      <div className="h-12 w-12 rounded-full bg-white" />
      {label ? <p className="text-sm text-slate-700">{label}</p> : null}
    </div>
  );
}
```

后续可以把它接到：

- `src/components/assessment/QuestionCard.tsx`
- `src/components/diagnose/DiagnoseResult.tsx`
- `src/components/video/VideoAnalysisResult.tsx`
- `src/components/plan/DayPlanCard.tsx`

### 动效上的注意点

- 动画一定要短，最好 1 到 2 秒循环
- 动画应该服务理解，不是装饰
- 要支持 reduced motion，避免对部分用户造成负担

相关样式入口：

- `src/app/globals.css`

## 我建议 Claude Opus 优先处理的顺序

如果 Claude 要接这个 redesign，我建议它按下面顺序做，而不是一次全改：

1. 首页 + 问题诊断  
   这是最影响第一印象和转化的部分。

2. 水平评估  
   直接减少题目数，能立刻降低使用门槛。

3. 视频诊断  
   这是最强功能，但也是最重页面，应该做分层。

4. 内容库 + 博主榜  
   它们更像目录页，可以稍后再收。

5. 训练计划  
   最后做“今日重点化”。

## 一句话总结

TennisLevel 现在的问题不是功能不够，而是首屏决策负担偏大。  
下一阶段最值得做的，不是继续加模块，而是把每个页面都收成：

- 更少的问题
- 更少的字
- 更强的主线
- 更像教练一句话带你往下走
- 更少“解释系统”，更多“直接帮你开始”
