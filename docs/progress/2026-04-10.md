## 今日核心进展
- 4 月 10 日这一天不是只做了 `PR8`，而是按 [[../../2026-04-10-implementation-plan]] 把主链连续推进了一整轮：`PR1` 到 `PR6` 都在当天实际落地，随后继续完成 `PR8`，并在最后把 diff 收成可 review 的 clean slice。
- 这一天的节奏很清楚：
  - 先把 diagnose / recommendation / plan 这条 consumer 主链按顺序收口
  - 再把 assessment 改成真正能给 plan 提供结构化 seed 的 `PlayerProfileVector`
  - 最后把 branch hygiene 当成 blocker 处理，避免把当天工作留成一团不可 review 的混合 diff
- 所以今天的进度应该理解成两层：
  - 当天实际做过的工作：`PR1-PR6 + PR8 cleanup`
  - 到收工时工作树里还保留的未提交代码：以 `PR8` 为主的 clean diff

## PR1. Taxonomy / contract 统一
- 今天起手做的是把 diagnosis、content、plan 之间最容易漂的 problem tag 和 contract 先收紧。没有这一步，后面 recommendation、plan handoff、assessment profile 都会继续建立在一堆半兼容别名和历史字段之上。
- `PR1` 的核心不是“重命名几个 tag”，而是把 canonical taxonomy 和 validation 真正变成共用基础设施。这样 diagnosis 给出的 problem、content 侧可检索的 tag、以及 plan 侧能吃进去的 weakness，不再各自维护一套近似但不完全相同的字典。
- 这一步当天的实际价值，是把后续 `PR2-PR8` 都放到了同一套可验证 contract 上。它本身不一定最显眼，但如果没有它，后面的 recommendation relevance、assessment handoff、plan blueprint 都会继续被旧 alias 和 pseudo tag 污染。

## PR2. Recommendation contract 收口
- 在 taxonomy 收稳之后，今天接着把 recommendation 从“排出来的内容列表”改成更像真正 product contract 的结构。推荐结果不再只是 `ContentItem[]`，而是带着对应 problem、推荐类型、为什么值得看等明确字段往下游传。
- 这一步最关键的判断，是 recommendation 不能继续只靠 loose ranking 暗示含义。diagnose 页面和 plan 页面都需要知道：这条内容为什么被选中，它解决的是哪个问题，它是技术解释、练习素材，还是比赛拆解。
- 同时今天也把机械类问题对 `match_breakdown` 的默认压制补上，让 recommendation 的内容类型更符合 diagnosis 的真实意图。这样 recommendation 不会再因为相似关键词，就把比赛评论类内容混进本该以技术纠偏为主的结果里。

## PR3. Diagnose recommendation / Library explore search 拆面
- `PR3` 解决的是一个很实际的体验混乱：诊断页里的推荐视频和库页里的主动搜索，本来是两种完全不同的行为，却长期共用一块混合 UI。
- 今天把它拆成了两套清楚的界面：
  - diagnose 页面只展示 diagnosis 派生出来的推荐搜索
  - library 页面保留用户主动输入、明确提交的 explore search
- 这个拆分很重要，因为 TennisLevel 的价值不是“让用户到处搜内容”，而是先 narrowing，再给一个合理方向。把 diagnose recommendation 和 explore search 混在一起，会削弱“系统先判断，再路由”的产品主线。今天这一步等于把 recommendation 和 search 的产品边界重新拉清楚了。

## PR4. Diagnose result surface 简化
- `PR4` 主要是在产品表达层面收刀。之前 diagnose result surface 层次太多，默认首屏里同时塞进原因、证据、补充解释、推荐内容和深层展开，用户第一眼并不容易抓住“我现在到底先练什么”。
- 今天把结果面改成更明确的两层结构：
  - 第一层只保留一个主要问题、一个短原因、一个直接动作和 plan CTA
  - 第二层才承接证据、深层解释、scene summary 和推荐内容
- 这一步的意义不只是 UI 更清爽，而是把 TennisLevel 的主张重新落回 “one primary next step”。如果默认首屏继续像报告页一样堆信息，系统虽然看起来“更完整”，但用户更难立刻抓到下一步动作。

## PR5. Standard / Deep diagnose 双深度产品化
- `PR5` 把 diagnose 从“只有一条默认路径”改成真正的双深度产品：`standard` 负责快速诊断，`deep` 则把现有 scene reconstruction 能力接到真实用户流里，而不是停留在模块或测试层。
- 今天这一步很关键，因为 deep mode 不应该是一个和主链脱节的技术演示。它必须仍然遵守同样的 diagnosis ownership：scene reconstruction 只负责把输入补成更结构化的场景，最终判断还是回到 deterministic diagnose core。
- 落地之后，`standard` 和 `deep` 的区别终于不只是 UI 文案，而是两种明确的输入深度。这样后续 `PR6` 的 bounded mediation 和 `PR8` 的 assessment seed，才有稳定的位置去接到 diagnose 主链上。

## PR6. Local bounded mediation 收口
- `PR6` 解决的是 diagnose 入口的“弱输入”问题。今天不是把 `/diagnose` 改成聊天助手，而是在 deterministic-first 前提下，加了一层只在必要时介入的 mediation。
- 这层 mediation 的边界今天被收得很紧：
  - clear complaint 直接跳过
  - noisy complaint 给短 paraphrase
  - genuinely vague complaint 最多追问一句
  - 再弱也不会演变成多轮 chat
- 同时，今天还把 local model、validator、fallback 和 telemetry 一起收进了同一个 bounded contract。这样 mediation 既是真实能力，不是空接口；又不会把 diagnosis ownership 偷偷迁移给模型。这个收口对于后面 assessment/profile/plan 主链也很重要，因为它保证了系统入口质量提高了，但产品边界没有被搞乱。

## PR8. Assessment 10+2 redesign
- 在前面 `PR1-PR6` 把主链打稳之后，今天后半段才继续推进 `PR8`。这部分不是把旧 assessment 多加几道题，而是把整条评估链从 coarse/fine branching 改成真正可复用的 profile generator。
- 最终落成的是：
  - 10 个 scored core questions
  - 2 个 lightweight profile questions
  - 只绑定前 10 题的 progress bar
  - `PlayerProfileVector` 作为结果 contract
  - assessment -> plan handoff 改成吃结构化 profile，而不是旧式 summary 拼接
- 这一步的价值在于，assessment 终于不再只是“给一个大概等级”，而是变成后续训练路线的稳定 seed。plan 侧能明确读到 level band、primary weakness、secondary weakness、play style 和 play context，而不是再从旧问卷残留字段里硬猜。

## PR8 cleanup. Branch hygiene / migration / verification
- 今天最后一段工作，不是继续往下做新功能，而是把 `PR8` 这条线收干净。因为到这个阶段，真正的风险已经不在 feature logic，而在 branch hygiene：如果 diff 里还混着 earlier PR residue、docs reorg、和其他无关改动，后面 review 和 merge 一定会出问题。
- 所以今天专门做了三件收口工作：
  - 把工作树清成 PR8-only diff，剔除 earlier PR 残留和无关 branch noise
  - 给旧 persisted assessment state 加最小 migration guard，避免 `profileVector` 缺失导致 restore flow 断掉
  - 重新跑 tests / build / diff checks，确认当前 slice 至少是 clean and reviewable
- 这个 cleanup 的意义很现实：它让“今天实际做了 PR1-PR6 + PR8” 和 “当前未提交还剩什么” 这两件事不再混成一团。到收工时，历史执行层面确实推进了整条主链，但工作树层面已经被收成一个可以单独 review 的 PR8 slice。
