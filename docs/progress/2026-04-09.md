## 具体变化
### 1. `/diagnose` 终于有了“只在必要时出现”的 mediation
- 今天最重要的产品判断不是“加一个 AI 层”，而是把 mediation 限制成一个非常窄的 intake helper：
  - clear complaint：完全跳过
  - noisy but recoverable complaint：给一条短 paraphrase
  - genuinely vague complaint：只追问一句
- 这意味着大部分本来就清楚的输入，例如：
  - `反手总下网`
  - `发球没信心`
  - `比赛里截击老下网`
  - `My overhead keeps going long`
  仍然保持一句话直达 diagnosis，不会被额外 UI 打断。

### 2. PR6 的边界真的落成了 typed contract，而不是页面上的 patch
- 早上先把 spec 和 plan 写清楚，下午真正落地时最关键的一点，是把 mediation 约束成一个小 contract，而不是让它渗进 diagnosis ownership：
  - `shouldMediateDiagnoseComplaint()`
  - `DiagnoseMediationResult`
  - 本地 validator
  - `prepareDiagnoseSubmission()` 里的 deterministic-first orchestration
- 尤其是 `clarificationUsed` 这次没有留在 UI 层当临时开关，而是被收进 typed intake contract，变成真正的 one-shot clarification 规则。
- 这件事很关键，因为它让“只追问一次”不是页面偶然行为，而是 orchestration contract 本身的一部分。

### 3. Clarify path 被收成 quiet inline flow，而不是 chatbot
- `/diagnose` 页面今天新增的是一个非常轻的 inline mediation block，而不是新会话状态：
  - paraphrase 时显示一条短 rewrite
  - clarify 时显示一句短问题 + 一个 free-text 输入
  - 回答后只 rerun 一次 intake
- 今天还专门修掉了第一版里最危险的 async 问题：
  - clarification rerun 不能 stale-overwrite 用户后续的新输入
  - clarification rerun 不能被记成第二次普通 diagnose submit
  - 用户改字或清空后，旧 clarification prompt 和旧 rerun 都要失效
- 这部分是今天真正的 implementation quality 收口点，不是 spec 里抽象写到就算完成。

### 4. Local model mediation 变成真实路径，但仍然保持 fallback-first
- PR6 不是只做“接口预留”。今天还把 mediation 默认接上了现有 local Qwen boundary，而不是永远停在 `model_unavailable`。
- 但这个接法仍然很克制：
  - 复用现有 `createLocalQwenClient()`
  - 新增一个非常窄的 diagnose mediation prompt
  - 只允许输出 `paraphrase / clarify / fallback`
  - 不允许 diagnosis、plan、advice 文本进入 contract
- 如果 model 不可用、输出非法、跨 category 漂移、或者低置信，整个链路仍然会静默退回 deterministic/raw path，不会把产品变成“AI 一挂就卡死”的流程。

### 5. Telemetry 也收口成 bounded trace，而不是偷记 raw text
- 今天还把 PR6 要求的 observability 一起补上了，但严格避开 raw complaint logging。
- 现在 diagnose mediation 这条链会记录：
  - gate decision
  - mediation mode
  - validator rejection reason
  - fallback reason
  - post-clarification outcome
- 这些 telemetry 是通过小的 observer/trace boundary 接进去的，没有把用户原始 complaint 塞进事件 payload，也没有为了埋点去污染 public mediation result contract。

## 与昨天和前几天的区别
- 4 月 8 日主要是在 Deep Mode completeness / category consistency 上继续补 diagnose 深链；今天则是把 focus 拉回 consumer `/diagnose` 的 free-text intake 入口本身。
- 如果说前几天在解决的是：
  - 深层场景信息不够
  - diagnosis -> plan 会断
  - category 会漂
- 那今天解决的是另一个更前置的问题：
  - 用户第一句话本身如果模糊，系统应该怎么“轻量帮他收口”，同时又不假装自己是聊天教练。
- 所以今天不是在扩 diagnosis 的“深度”，而是在补 diagnosis 的“入口质量”和“边界诚实度”。

## 仍然存在的边界
- PR6 仍然是 bounded mediation，不是更泛化的 LLM intake assistant：
  - 没有 live mic
  - 没有多轮对话
  - 没有 assistant 直接给 diagnosis / plan
  - 没有新的 provider abstraction surface
- local model 路径虽然已经是真实路径，但仍然故意保守；这不是智能上限，而是产品边界选择。
- `npm run lint` 仍不是一个稳定验证入口，因为 repo 还会触发 Next 的 ESLint 初始化交互。
