## 今日核心进展
- 把场景还原从独立 prototype 逐步收口到 `/diagnose` 主链，明确 `/diagnose` 是 deep flow 的唯一 orchestrator。
- 修通了场景还原 -> diagnosis -> plan 的结构化 handoff，不再只靠原始文本或 `effortMode=deep`。
- 把训练计划从“前半周具体、后半周泛化”推进到 deterministic deep overlay，至少在 `serve` gold slice 上做到 Day 1 到 Day 7 都保持场景特异性。
- 补了一组 deterministic regressions，把 “deep 真的能深到 Day 7” 锁住，优先防止 late-week decay 回退。

## Diagnosis 方向的推进

### 1) 场景还原原型链路成形
- 新增 `scenarioReconstruction` 命名空间、question bank、selector、runtime、LLM thin boundary、API routes。
- 场景还原最小闭环已跑通：
  - 自由文本输入
  - 结构化场景提取
  - follow-up 追问
  - 完成态 CTA
  - handoff 回到现有 `/diagnose`

### 2) completion-state handoff 接回现有 diagnosis 主链
- 不再让场景页拥有第二套 diagnosis/plan pipeline。
- 新增纯函数 handoff adapter，把 `ScenarioState` 转成 diagnosis-friendly input。
- `done = true` 时会进入现有 `/diagnose?q=...`，之后的 diagnosis result、snapshot persistence、study artifact persistence 仍由 `/diagnose` 统一负责。

### 3) `done` 语义收紧
- `done = true` 时：
  - `selected_question = null`
  - `eligible_questions = []`
  - `scenario.next_question_candidates = []`
  - `scenario.selected_next_question_id = null`
- 允许保留少量 non-blocking missing slots，但不再驱动追问。

### 4) serve / second-serve 相关诊断问题修复
- 修复了场景还原后 handoff 到 `/diagnose` 时把 “二发下网” 错写成 “一发发不进” 的 bug。
- 现在 `二发 / 第二发 / second serve / second-serve / secondserve` 会被稳定归一化到 serve family，并在 handoff 中保留二发语义。
- 对于 `关键分 + 二发 + 下网` 这类单一稳定问题，direct CTA 与 downstream diagnosis 能正确接通。

## 从 prototype 到 deep orchestrator

### 5) `/diagnose` 成为 deep flow 的唯一 orchestrator
- `quick / standard / deep` 保留在同一诊断容器里。
- 选择 `deep` 后，场景还原不再是主要平行页面，而是 `/diagnose` 内的 inline subflow。
- deep state 的进入、完成、回退、重置都收回到 `/diagnose` 容器状态里管理。
- `/diagnose/scenario` 逐步降级为 bridge/debug route，不再是主用户旅程。

### 6) 引入 `EnrichedDiagnosisContext`
- 新增 deterministic 结构化 handoff，对 deep diagnosis 与 deep plan 共用：
  - `mode`
  - `sourceInput`
  - `sceneSummaryZh / sceneSummaryEn`
  - `problemTag`
  - `level`
  - `strokeFamily`
  - `serveSubtype`
  - `sessionType`
  - `pressureContext`
  - `movement`
  - `outcome`
  - `incomingBallDepth`
  - `subjectiveFeeling`
  - `isDeepModeReady`
- 该对象已进入 diagnosis session，并贯穿 diagnosis -> plan handoff。

### 7) 保持单一 DiagnoseResult 面
- 没有拆第二套 deep result page。
- 仍然保留一个主 diagnosis result surface。
- deep 模式下只追加：
  - deep badge
  - 简短 scene recap
  - “为什么这次判断更具体” block
  - deep-specific plan CTA
- serve vertical slice 现在已经能清楚表达：
  - 普通诊断：基于原始文本
  - 深入诊断：基于原始文本 + 场景还原证据

## Training Plan 方向的推进

### 8) deep context 真正进入 `/plan`
- `/plan` 不再只浅层依赖：
  - `problemTag`
  - `level`
  - `preferredContentIds`
  - `sourceType`
  - `primaryNextStep`
- diagnosis -> plan handoff 现在会保留 structured `deepContext`，避免退化回 raw text 或 `effortMode=deep` 单独决定。

### 9) 7 天计划从浅层模板推进到 deep overlay
- 没有新起第二套 plan page，也没有引入 runtime LLM 计划生成器。
- 仍以现有 base plan 为起点，叠 deterministic deep-mode overlay。
- 当前至少在 `serve` gold slice 上，deep plan 已重写全 7 天，而不是只改 summary 或 Day 1。

### 10) PR4: 重点消 late-week decay
- serve deep plans 现在按明确 progression ladder 推进：
  - Day 1: baseline / clean pattern
  - Day 2: stabilize
  - Day 3: add key scene variable
  - Day 4: timing / feeling variability
  - Day 5: pressure
  - Day 6: transfer / serve-plus-one sequence
  - Day 7: consolidate + carry-forward rule
- 深层 scene variables 已开始显式驱动每日内容：
  - first/second serve
  - key-point pressure
  - outcome
  - subjective feeling

## Quality lock / regressions

### 11) PR5: 用 deterministic golden fixtures 锁 deep plan 质量
- PR5 不是新 feature slice，而是 quality-lock slice。
- 新增/强化 deterministic fixtures，防止 deep plan 在后半周 silently collapse back to generic template language。
- 目前以 `serve` 作为 gold regression slice，优先锁：
  - Day 4 必须仍是 timing/feeling variability
  - Day 5 必须仍是 pressure
  - Day 6 必须仍是 transfer sequence
  - Day 7 必须仍是 carry-forward rule
