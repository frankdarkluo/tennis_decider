## 今日核心进展
- 今天早上真正补上的，不是再写一层 Deep Mode 原理，而是把 Deep Mode 在 `serve` 这条链上从“方向对了”推进到“链路更闭合”。
- 两份 spec 已经定义了大的边界：
  - `2026-04-08-deep-mode-completeness-design.md` 解决 Deep Mode 不该太早 handoff
  - `2026-04-08-deep-mode-category-consistency-design.md` 解决 Deep Mode category 不能在下游漂移
- 但今天早上真正新增、且不只是重复 spec 的部分，主要有三块：
  - 把 `serve` 的 Deep Mode 从抽象 required slot 落成可问、可答、可传递的发球专属线索
  - 把“结果页还说缺线索”改成能回流到 Deep Mode 的真实链路，而不是静态提示
  - 修掉 diagnosis -> plan 的致命断链：从诊断页点训练计划 CTA 会被 `/plan` 页旧 assessment gate 错误打回 `/assessment`

## 具体变化
### 1. Serve Deep Mode 终于不再只是“知道这是发球”
- 在 spec 里，Deep Mode completeness 主要定义的是 category-aware required/optional completion 边界；今天早上真正落地的是 `serve` 的可操作化细节。
- 新增了两类发球专属 Deep Mode 信息：
  - `serve.control_pattern`
    - `net / long / wide / no_rhythm / unknown`
  - `serve.mechanism_family`
    - `toss / contact / rhythm / direction_control / unknown`
- 这意味着系统现在不只是知道“这是发球问题”，还会继续问：
  - 更像下网、偏长、偏左偏右，还是完全没节奏
  - 更像抛球不稳、击球点乱、节奏乱，还是方向控制不住
- 对应地新增了两道发球专属 follow-up：
  - `q_serve_control_pattern`
  - `q_serve_mechanism_family`
- 这部分是今天早上的实质推进，因为之前 spec 只定义了 completion 和 category gate，并没有把 serve-specific family 真正 operationalize 到 question bank / answer model / handoff 文本里。

### 2. 真实用户说法被补进了 deterministic parser
- 今天早上还补了一个很关键但容易被忽略的点：系统以前只认“`不受控`”，不太认真实用户更常说的“`不太受控 / 控制不住`”。
- 现在这些口语表达会被收敛到 `outcome.primary_error = no_control`，不会再因为表述太日常而低估问题。
- 这类修补不在两个 spec 里，但对实际体验影响很大，因为用户给的不是 schema 友好的标注语句，而是“发坏了、控制不住”这种球场口语。

### 3. Deep Mode -> diagnosis 的 handoff 文本更像真正能被下游吃懂的线索
- 两份 spec 都讲了 handoff contract 和 category gate，但今天早上还做了一个更落地的收口：把新的 serve 内部线索写回 diagnosis handoff 文本和 enriched context。
- 现在 handoff 不只是：
  - 发球
  - 比赛/练习
  - 紧张
- 还会补进：
  - 发球更像总下网 / 总偏长 / 偏左偏右 / 完全没节奏
  - 更像抛球 / 击球点 / 节奏时机 / 落点方向控制
- 同时补了 `diagnosisRules` 里的 serve 同义词，让这些 Deep Mode 线索更容易被下游规则真正吃到，而不是结构上有 handoff、词面上却接不上。

### 4. 结果页“缺线索”不再只是静态中间态
- 之前的问题是：
  - Deep Mode 太早 handoff
  - 结果页又说“先补一条关键线索”
  - 用户被卡在一个半结果、半提醒的中间态
- 今天早上在这个基础上补了一层真实回流：
  - 诊断页应用 Deep Mode handoff 时，不再用内部 summary 覆盖原 complaint input
  - 这样 DeepScenarioModule 不会因为 input 变化而把自己的场景状态清空
  - 当结果页仍然是 `needsNarrowing` 且处于 deep flow 时，UI 现在会给“继续补场景线索”CTA，而不是只留一句静态提示
- 这部分不是 spec 的大边界，而是把 spec 里的“honest handoff / keep deep mode semantics separate”真正收成可走通的页面链路。

### 5. 旧场景对象兼容性被补了，不会因为新 serve 子结构直接炸掉
- 早上还顺手修了 runtime 的一个兼容性问题：旧的 `ScenarioState` fixture 或持久化数据可能没有新加的 `serve` 子结构。
- 现在 `ensureScenarioInternals()` 会自动补齐缺失的默认字段，再重算 progress，而不是假设所有旧对象都已经是最新 schema。
- 这类变化不在 spec 里，但很关键，因为 Deep Mode 这轮修改已经跨过了 question bank、route、page state、draft/handoff，多一处 schema 假设就容易在老数据上出问题。

### 6. Diagnosis -> plan 这条链又补了一个产品级断点
- 这件事不在今天这两份 deep-mode spec 里，但它是今天早上最明显的真实用户 bug 之一：
  - 在 diagnose 结果页点击“生成更具体的 7 步训练计划”
  - URL 本来已经是 diagnosis-origin 的 `/plan?...source=diagnosis`
  - 但 `/plan` 页自己还保留了旧的非 study assessment gate
  - 页面一加载就把用户打回 `/assessment`
- 今天把这个 gate 从 `/plan` 页上移掉了：
  - study mode 的 `pendingStudySetup` gate 仍保留
  - diagnosis-origin 的 plan 现在可以直接打开
  - 直接空开 `/plan` 会落到安全空状态，而不是被误重定向到评估页
- 这部分和 Deep Mode 不是同一层 abstraction，但它直接影响 diagnose -> plan 主链是否闭合，所以今天一起收口是合理的。

### 7. UI 界面的交互重构与状态收拢
- 除上述深度模式的逻辑链路补齐外，今晚大面积修复并清理了关键界面的交互规范与认知负担，使其完全对齐「克制展示、强制确认」的产品法则：
  - **诊断结果页 3 层结构拍平 (DiagnoseResult)**：摒弃了之前繁琐的渐进式多层交互，默认只显示最重要的结论、动作标签与行动按钮区，将繁杂的原因分析全部收纳进 "包含原因与深层信息" 折叠面板中供按需翻阅。
  - **斩断错误的前置预测和缓存穿透 (Diagnosis State)**：修复了诊断输入框实时预测引发的历史残留 UI 问题，目前通过新增强制的 `hasCommitted` 锁，确保修改深度、点选案例 tag 甚至单纯在输入框打字都只会暂存参数并**隐藏历史结果**，使用户只有在明确点击「开始诊断」后才能看到系统判定，所见即所得。
  - **内容库 live-query 断链 (Library Search)**：破除了原来的输入框内容联想“边打字边过滤”行为，分离了临时输入的 `keyword` 与实际生效的 `committedKeyword`。并在过滤栏旁边显式增加了一个专门的“搜索”按钮，进一步控制信息的涌入频率。
  - **修掉了从博主榜强制跳转评估的 Hydration Bug**：修正了 `rankings` 页面中 `storedAssessmentExists` 的异步状态初始值陷阱（由初始的 false 改为正确的 null），终于解决了页面第一次挂载期间误判其没有测过能力而无故把人弹出到 `/assessment` 评估页的 bug，让顶部导航栏的“博主榜”完全恢复点击功能。

## 与总体设计的关系
- `deep-mode-completeness` spec 解决的是“不要最小够用就放行”；今天早上额外推进的是：把 `serve` 真正补到“可问、可答、可传、可用于下游”的细粒度线索，而不是停在抽象 required slot。
- `deep-mode-category-consistency` spec 解决的是“不要从 serve 漂到 forehand”；今天早上额外推进的是：让 handoff 的词面和规则层也更对齐，减少“结构上同类、文本上太弱”的半断链状态。
- 两份 spec 都没有覆盖 diagnosis -> plan 的 assessment redirect bug；这个是今天早上新增发现并修掉的链路问题。

## 当前状态
- 发球这条 deep diagnosis 链现在比昨天更接近产品想要的样子：
  - 不再轻易跨技术类别
  - 不再只凭浅层信息就说“场景已经补全”
  - 结果页也不再只是告诉用户“你还缺线索”，而是开始具备回到 Deep Mode 继续补线索的能力
- diagnosis -> plan 也不再被旧 assessment gate 打断，主 CTA 至少能落到正确页面上。

## 仍然存在的边界
- 今天真正被深化的是 `serve`；`return / volley / overhead / slice` 还没有被同等粒度地 operationalize 到“专属 family + 专属线索 + 下游桥接”这一级。
- 结果页目前已经有回流 CTA，但还没做到“下游明确缺哪个 slot，就自动驱动 Deep Mode 继续问那一题”。
- `app-smoke` 里还有少量旧断言写死了老模板文案，这不是今天的主 bug，但后面最好继续清。
